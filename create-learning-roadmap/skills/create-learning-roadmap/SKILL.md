# 学習ロードマップ & 問題集 Issue 作成スキル

## 概要

指定トピックの学習ロードマップを初級・中級・上級で設計し、各レベルを複数 Phase に分割した問題集を GitHub Issue として作成する。
特訓級はユーザーからの個別依頼があったときだけ追加する。
外部リソース（URL・書籍名・サイト）は含めない。

## レベル定義

| レベル   | 内容 |
|----------|------|
| 初級     | 構文・基本概念・型 |
| 中級     | 設計・エラー処理・標準ライブラリ・テスト |
| 上級     | 並行処理・パフォーマンス・実践設計 |
| 特訓級   | ユーザーが不安・知識不足を補うために個別依頼する特化問題集（特定の設計パターン・難所・イディオム）。1 Phase = 1 テーマ |

## Issue 階層構造

3 層の親子構造で管理する。GitHub の sub-issues API でレベル Issue → Phase Issue を紐付ける。

```
<トピック>          ← 親 Issue（例: "Golang"）
  ├── 初級          ← レベル Issue
  │     ├── [Phase 1] <サブトピック>   ← Phase Issue
  │     └── [Phase 2] ...
  ├── 中級
  │     └── [Phase 1] ...
  ├── 上級
  │     └── [Phase 1] ...
  └── 特訓級         ← レベル Issue（依頼があったときだけ作成）
        └── [Phase 1] <テーマ名>   ← Phase Issue
```

- Phase Issue のタイトルに `[初級]` などのレベル接頭辞は**付けない**（階層で表現する）
- Phase 番号はレベルごとにリセットする（初級 Phase 1〜N、中級 Phase 1〜M）

## ワークフロー

### 通常ワークフロー（初級・中級・上級）

1. トピックを確認（引数があればそのまま使用、なければ AskUserQuestion で確認）
2. 各レベルのサブトピックを洗い出し、Phase 単位に分割する（「Phase 設計」参照）
3. トピック Issue を作成する
4. 初級・中級・上級の**レベル Issue**を作成し、トピック Issue の子に紐付ける
5. 各 Phase Issue を作成し、対応するレベル Issue の子に紐付ける
6. 作成した Issue の一覧をチャットに出力する

特訓級は通常ワークフローでは作成しない。

### 特訓級ワークフロー（個別依頼があったときだけ実行）

トリガー: ユーザーが「〇〇の問題を作って」のように、特定テーマを名指しで依頼したとき。

1. トピック Issue が存在するか確認。なければ作成
2. `特訓級` レベル Issue が存在するか確認
   - **なければ作成**し、トピック Issue の子に紐付け
   - あればそれを使う
3. 依頼テーマで Phase Issue を作成（タイトル: `[Phase N] <テーマ名>`、N は特訓級内の既存 Phase 数 + 1）
4. Phase Issue を特訓級レベル Issue の子に紐付け

## Phase 設計

### Issue 数と問題数の目安

**1 Issue = 1 サブトピック。** サブトピックを細分化して Issue を分割することで、体系的・網羅的な問題集を作る。

| レベル   | 推奨 Issue 数（サブトピック数） | 1 Issue あたりの問題数 |
|----------|------------------------------|----------------------|
| 初級     | 7〜9                         | 10〜30問              |
| 中級     | 6〜8                         | 10〜30問              |
| 上級     | 5〜7                         | 10〜30問              |
| 特訓級   | 依頼があったテーマだけ追加     | **最低 15問**（上限なし）|

### サブトピックの粒度

**1 Issue = 単一の概念・機能領域**。複数の概念をまとめた題目は分割する。

| 悪い例（複合）                          | 良い例（単一）                                |
|---------------------------------------|---------------------------------------------|
| 変数・型・制御フロー (1 Issue)          | 変数と宣言 / 型システム / 制御フロー（3 Issues）|
| インターフェース・型アサーション (1 Issue) | インターフェース / 型アサーション（2 Issues）   |

### サブトピック設計の手順

1. そのトピックの公式ドキュメントの目次・章立てをサブトピックの出発点にする
2. 「1 つの問題文で説明できる最小単位」まで分割する
3. レベル定義（構文 / 設計 / 実践）に当てはめてレベル分けする

## 問題の書き方

Issue は採点用ではなく、**手を動かしながら概念を習得するための学習教材**として設計する。
基本方針: ゼロから全部書かせない。大枠のコードを提示し、欠損部分だけ補完させる。

### 問題タイプ（4 種類）と構成比の目安

| タイプ          | 説明                                                       | 目安  |
|---------------|----------------------------------------------------------|------|
| 穴埋め         | 骨格コードの `___` 欠損箇所を埋めさせる。1 問につき 1〜3 箇所  | 50%  |
| メソッド実装    | 型・シグネチャ・コメント済みのスタブ本体を実装させる            | 25%  |
| コンパイル修正   | 壊れたコードを渡し、修正してコンパイルを通させる                | 15%  |
| 比較・読み解き   | 動くコードを 2 つ提示し「どちらが正しいか・なぜか」を答えさせる   | 10%  |

### 統一フォーマット（全タイプ共通）

```text
### 問題N: <項目名>（<タイプ名>）
<課題文（1〜2 文）>

```<言語>
// 出題コード
//   穴埋め: 欠損箇所を ___ または // TODO: <何を書くか> で示す
//   メソッド実装: 本体に // TODO: 実装する を置く
//   コンパイル修正: エラー行に // ← エラー コメントを付ける
//   比較: 2 つのスニペットを並べる
```

<details>
<summary>解答例</summary>

```<言語>
// 解答コード
```

**ポイント:** <重要な概念・落とし穴・なぜそうなるか（1〜3 行）>

</details>
```

### 問題集全体のルール

- URL・書籍名・外部サイトは含めない
- 問題は難易度順（易しい → 難しい）に並べる
- 解答例は `<details><summary>解答例</summary>` で折りたたむ。`</details>` の直前に空行を入れる（GitHub の折りたたみ解釈のため）
- 解答例には **ポイント:** 行で重要な概念・注意点を 1〜3 行で補足する
- 解答例は模範的な書き方を 1 つ選ぶ（言語らしいイディオムを優先）
- 各 Issue 末尾に**チェックリスト**を付ける（「すべての問題を解いた」「解答例と比較した」「概念を自分の言葉で説明できる」）

### コンパイル修正問題の補足ルール

- **1 問 1 エラー**を基本とする（複数エラーは中級以上）
- **ポイント:** にはエラーの根本原因を書く（「型が違う」だけでなく「なぜその型制約があるか」まで）
- レベル別の代表的なエラー種別:

| レベル | エラー種別 |
|--------|-----------|
| 初級   | import 漏れ / 未使用 import・変数 / 型の不一致 / 構文エラー / return 漏れ |
| 中級   | インターフェース未実装 / 型アサーション失敗 / エラー未処理 / nil 参照 |
| 上級   | 型制約違反（ジェネリクス） / 並行処理リーク / deadlock / 低レベル誤用 |

## Issue 作成（実行手順）

### ラベル作成

Issue 作成前に必要なラベルが存在しなければ作成する:

```bash
gh label create "学習" --color "#0075ca" --description "学習用 Issue" 2>/dev/null || true
gh label create "<トピック>" --color "#e4e669" --description "<トピック>学習" 2>/dev/null || true
gh label create "初級" --color "#d93f0b" 2>/dev/null || true
gh label create "中級" --color "#e99695" 2>/dev/null || true
gh label create "上級" --color "#b60205" 2>/dev/null || true
gh label create "特訓級" --color "#5319e7" --description "個別依頼の特訓問題集" 2>/dev/null || true
for i in $(seq 1 9); do
  gh label create "phase-$i" --color "#0052cc" --description "Phase $i" 2>/dev/null || true
done
```

### Phase Issue 本文テンプレ

`<問題N>` には「問題の書き方 → 統一フォーマット」のブロックを問題数だけ繰り返し並べる。

```bash
gh issue create \
  --title "[Phase N] <フェーズ名>" \
  --body "$(cat <<'ISSUE_EOF'
## 学習目標
<このフェーズで習得すること>

## 問題一覧

<問題1（穴埋め）>
<問題2（メソッド実装）>
...
<問題N（コンパイル修正・比較など）>

## チェックリスト
- [ ] すべての問題を解いた
- [ ] 解答例と自分の実装を比較した
- [ ] 概念を自分の言葉で説明できる
ISSUE_EOF
)" \
  --label "学習,<トピック>,<レベル>,phase-N"
```

### sub-issues 紐付けコマンド

```bash
# Issue の数値 ID を取得
gh api repos/<owner>/<repo>/issues/<child_number> -q '.id'

# 子 Issue を親 Issue に紐付け（-F で integer 型として送信）
gh api --method POST repos/<owner>/<repo>/issues/<parent_number>/sub_issues \
  -F sub_issue_id=<child_issue_integer_id>
```

### heredoc の注意点

`gh issue create --body "$(cat <<'EOF' ... EOF)"` でコードブロックを書くときの落とし穴:

| 状況 | 対処法 |
|------|-------|
| 本文中にバッククォート 3 つを使いたい | heredoc 内で ` ``` ` はそのまま書ける。**ただしデリミタ（EOF）と同じ文字列が行頭に来ると heredoc が終わってしまう**ので、`'ISSUE_EOF'` など固有名にする |
| `<details>` 内のコードブロックが折りたたまれない | `</details>` の直前に空行を入れる |
| `$変数` が展開されてしまう | デリミタを `'EOF'`（シングルクォート）で囲んで展開を抑制する |
